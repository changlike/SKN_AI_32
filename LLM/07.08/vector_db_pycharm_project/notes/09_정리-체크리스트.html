정리 & 체크리스트

15강에서 배운 것

15강에서는 벡터를 영속 저장하고 대량·고속으로 검색하는 벡터DB를 배웠습니다.

  • 벡터DB의 역할: 인덱싱(사전 1회)과 서비스(매 요청)를 분리.
  • FAISS vs Chroma: FAISS는 가볍고 빠름, Chroma는 메타데이터 필터에 강함.
  • 영속화 아키텍처: 요청 핸들러에서 재인덱싱 금지, build/service 분리.
  • 메타데이터 필터: 특정 문서 안에서만 검색(Chroma).
  • 증분 업데이트: 새 문서만 add_documents로 덧붙임.

한눈에 보는 핵심 코드

from langchain_community.vectorstores import FAISS
from common import get_embeddings, DATA

emb = get_embeddings()

# === 인덱싱 (배치, 1회) ===
vs = FAISS.from_documents(chunks, emb)        # 임베딩 + 인덱스 (비쌈)
vs.save_local(str(DATA / "faiss_index"))      # 저장

# === 서비스 (서버 기동 시 1회 로드) ===
vs = FAISS.load_local(str(DATA / "faiss_index"), emb,
                      allow_dangerous_deserialization=True)   # 재임베딩 X
results = vs.similarity_search("질문", k=3)    # 빠른 검색

# === 증분 추가 ===
vs.add_documents(new_chunks)                   # 새 청크만 추가

이 강에서 설치한 라이브러리

pip install faiss-cpu langchain-community langchain-google-genai pypdf
pip install langchain-chroma   # (메타데이터 필터, 07번)

RAG 5단계 진행 상황

① 로드 ✓ · ② 청킹 ✓ · ③ 임베딩 ✓ · ④ 저장/검색 ✓(15강) · ⑤ 생성 ← 16강(다음)

이제 RAG의 검색 인프라가 완성됐습니다. 다음 강에서 검색 결과로 LLM이 답하면 RAG가 완성됩니다.

체크리스트

  • [ ] 벡터DB가 인덱싱과 서비스를 분리하는 이유를 안다.
  • [ ] FAISS와 Chroma의 차이를 설명할 수 있다.
  • [ ] from_documents(인덱싱)와 load_local(서비스)을 구분해 쓴다.
  • [ ] save_local/load_local로 영속화하고 재임베딩을 피했다.
  • [ ] 메타데이터 필터와 증분 업데이트를 적용할 수 있다.

한 줄 핵심

벡터DB는 인덱스를 미리 만들어 저장(영속화)하고, ANN으로 빠르게 검색한다. "비싼 임베딩은 한 번, 검색은 매번"이 RAG 인프라의 핵심이다.

다음 강 예고

15강까지 RAG의 검색 인프라(로드→청킹→임베딩→저장/검색)가 완성됐습니다. 16강 RAG QA 시스템에서 마지막 ⑤ 단계 — 생성 — 을 더해 RAG를 완성합니다. 검색한 문서를 근거로 LLM이 답하되, "문서에 없으면 모른다" 고 환각을 억제하고, 출처를 함께 표시하는 실전 QA 시스템을 만듭니다.
